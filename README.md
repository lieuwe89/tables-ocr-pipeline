# Groningen Adresboek 1926

Pipeline + web app for extracting and exploring the 1926 Groningen address
book. ~900 scanned pages → structured data with word-level bounding boxes →
CRM for human corrections → public map-centric explorer (planned).

**Status:** pilot pipeline run (838/838 pages, 60,783 entries, $7.49 LLM cost);
admin CRM functional; public-site work in progress (geocoding + SQLite + map +
deep-zoom viewer scheduled). See `NEXT_STEPS.md` for the live roadmap.

## Setup

### Cross-platform Python (macOS / Linux / Windows)

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt           # macOS / Linux
# .venv\Scripts\pip install -r requirements.txt     # Windows PowerShell
```

`requirements.txt` covers the LLM + alignment + export stages. The Surya OCR
package (`surya-ocr`) is **not** in `requirements.txt` because it is heavy
(PyTorch, ~3 GB) and only needed when running fresh OCR. Install it
explicitly when you need it:

```bash
.venv/bin/pip install surya-ocr "transformers<5"
```

### AMD GPU Support (Windows)
To use your AMD Radeon GPU on Windows, you should install `torch-directml`:

```powershell
.venv\Scripts\pip install torch-directml
```

Then in `pipeline/config_local.py`, set:
```python
OCR_DEVICE = "directml"
```

### API keys

Create `pipeline/config_local.py` (gitignored):

```python
OPENROUTER_API_KEY = "sk-or-v1-..."
OPENROUTER_MODEL   = "google/gemini-2.5-flash-lite"
LLM_PROVIDER       = "openrouter"   # or "google" for Google AI Studio direct
```

OpenRouter pay-per-token (~$8 for the full book) is the practical default;
Google AI Studio's free tier is capped at 20 req/day on Gemini 2.5 Flash.

### Web app

```bash
cd web
npm install
cp .env.local.example .env.local    # set ADMIN_PASSWORD
npm run dev                         # http://localhost:3001
```

Public route: `/page/<stem>` (read-only). Admin: `/admin/page/<stem>`
(basic-auth). Stats: `/admin/stats`.

## Pipeline usage

```bash
# Full pipeline (Surya OCR + LLM + alignment + export)
.venv/bin/python pipeline/run_pipeline.py

# Test slice
.venv/bin/python pipeline/run_pipeline.py --test 3 -v

# Specific page range
.venv/bin/python pipeline/run_pipeline.py --pages 150-160

# Surya OCR only (skip LLM)
.venv/bin/python pipeline/run_pipeline.py --ocr-only

# Force reprocess (ignore checkpoint)
.venv/bin/python pipeline/run_pipeline.py --reprocess
```

### Refresh outputs without re-OCR / re-LLM

After a fix to alignment or bbox repair logic, regenerate `output/json/`,
`output/alto/`, and `output/combined/` from cached OCR + cached LLM raw — no
network, no GPU:

```bash
.venv/bin/python pipeline/refresh_outputs.py
```

~14 seconds for all 838 pages.

## Output

| Format | Location | Purpose |
|---|---|---|
| Surya cache + word dump | `output/hocr/` | `<stem>.ocr.json` (cached) + `<stem>.words.txt` (debug) |
| LLM raw responses | `output/llm_raw/` | Per-page raw JSON for crash recovery + cheap re-export |
| ALTO XML | `output/alto/` | Standardized OCR interchange format |
| Per-page JSON | `output/json/` | Aligned entries with bboxes (web-app source) |
| Combined indexes | `output/combined/` | Search / address / street / cross-reference / page-manifest indexes |
| LLM usage sidecars | `output/llm_usage/` | Per-page token counts for cost reporting |
| Failure sidecars | `output/failures/` | Structured error records when retries exhaust |
| User overrides | `output/overrides/` | Admin CRM edits (this dir IS tracked in git) |

## Long unattended runs

Surya CPU OCR is the bottleneck — figure ~2 minutes/page on a modern 6-core
desktop CPU, or ~24–30 hours for the whole book. A few things matter for
unattended runs:

### Detach the process from the shell

A pipeline started from a foreground shell dies when that shell exits — and
that includes closing a terminal or ending an editor/agent session.

**Windows:**

```powershell
$proj = "C:\path\to\groningen-adresboek-1926"
$out  = "$proj\output\logs\pipeline_$(Get-Date -Format 'yyyyMMdd_HHmmss').stdout.log"
Start-Process -FilePath python -ArgumentList "-u","pipeline\run_pipeline.py" `
  -WorkingDirectory $proj -WindowStyle Hidden `
  -RedirectStandardOutput $out -RedirectStandardError "$out.err"
```

**macOS / Linux:**

```bash
nohup .venv/bin/python -u pipeline/run_pipeline.py \
  > output/logs/pipeline_$(date +%Y%m%d_%H%M%S).log 2>&1 &
```

### Disable OS sleep

Detachment survives shell exit, **not** OS suspend. Disable sleep for the
duration of the run.

### Two workers ≈ 1.6–1.8× faster

Surya is single-process; running `pipeline/ocr_worker_reverse.py` in parallel
processes pages from the back of the list. Both share `output/hocr/` as a
cache and meet in the middle. Cost: ~3 GB extra RAM. See `CLAUDE.md` →
"Speeding up the OCR pass" for the full launch incantation on Windows.

### Resumability

OCR caches per page; LLM raw responses persist per page; checkpoint tracks
completed/failed pages. Re-running picks up where it left off. Use
`--reprocess` for a clean rerun.

## Cost

- **OCR (Surya, CPU):** free; ~24–30 h on a modern 6-core CPU (~half that
  with the dual-worker setup). GPU would cut to ~30 min.
- **LLM via OpenRouter (gemini-2.5-flash-lite):** ~$0.01/page; full book
  (838 pages) was $8.75 in the pilot.
- **PDOK geocoding:** free; ~2.5 h with 20 concurrent workers.
